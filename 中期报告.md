# Infrastructure & Milestone Summary: Midterm Report

## 1. 架构总览 (System Architecture)

本项目旨在构建一个高可用、可容忍灰色故障（Gray Failures）的分布式推理系统。当前系统已成功完成基础架构的搭建与联通，整体架构划分为清晰的三层拓扑结构，确保了控制平面与数据平面的解耦：

- **负载均衡层 (Ingress - Gateway/Load Balancer)** : 部署于控制节点 VM 01 (`sp26-cs525-0601.cs.illinois.edu`)。该层作为整个集群的中心网关，负责接收客户端的推理请求，并基于后端的健康状态执行流量转发与负载均衡。
- **监控控制层 (Control Plane Daemon)** : 同样部署于控制节点 VM 01。核心组件为自定义的 `graypulse_daemon.py`。该守护进程作为集群的“观察者”与“决策者”，旁路运行于负载均衡层之上，负责高频次采集集群指标，并执行动态的节点状态判定。
- **基础设施层 (Backend - Inference Nodes)** : 由两个同构的计算节点 VM 05 (`gpu1`) 和 VM 06 (`gpu2`) 组成。作为数据平面的工作节点，承载实际的模型推理服务。

### 数据流向与控制逻辑

- **控制流 (Control Flow)** : 监控控制层的 Daemon 定期通过 Unix Socket 接口查询 Ingress 层 HAProxy 的后端存活状态，并实施精确的在线节点甄别与统计。
- **数据流 (Data Flow)** : 客户端外部请求抵达 VM 01 的暴露端口后，HAProxy 根据预设算法及存活状态表，将 HTTP/GRPC 请求透明代理至 `gpu1` (05) 或 `gpu2` (06) 的推理服务暴露端口。

---

## 2. 核心技术栈与网络配置 (Technology Stack & Networking)

为了保障系统的高性能与可扩展性，我们在每个层级均采用了行业标准的开源原生技术栈：

- **HAProxy (High Availability Proxy)** : 作为 Ingress 层的核心载体，配置为无缝反向代理与健康检查中心。通过暴露本地 Socket 文件 (`/var/run/haproxy.sock`)，结合强大的 `socat` 工具，实现了高效、低延迟的进程间通信 (IPC)，暴露底层统计拓扑。
- **Triton Inference Server** : 作为 Backend 层的统一推理引擎。现阶段采用 `nvcr.io/nvidia/tritonserver:24.05-py3` 镜像。为克服集群临时硬件（GPU配额）不可用的困境，通过定制启动参数（如 `--allow-gpu-metrics=false` 与 `--allow-cpu-metrics=true`），成功在 CPU 兼容模式下启动了全功能的 Triton 实例。
- **Python 驻留式守护进程 (Persistent Daemon Deployment)** : 利用 `nohup` 结合无缓冲模式 (`-u` 参数) 实现了监控进程与 SSH 会话生命周期的解绑，确保其作为常驻后台服务 (Background Daemon) 持久运行。标准输出与错误被安全重定向合并至 `daemon.log`，构建了完整、不间断的日志审计闭环。
- **网络连通性与端口映射 (Networking Validation)** : 定义 Triton Server 标准推断端点为 TCP 端口 `8000`。针对后端健康验证，在应用层构建了 `/v2/health/ready` 端点的细粒度监控，严格校验 HTTP 响应码 `200 OK`，由此 HAProxy 将确认为“可用 (Healthy)”，并纳入负载池。

---

## 3. 已完成的核心里程碑 (Key Milestones Achieved)

在系统从设计向代码落地的过程中，我们针对多项工程障碍实施了有效攻坚：

- **异构环境的隔离与连通 (Environment Mapping & Connectivity)** : 化解了本地开发机与远端集群间复杂的认证/网络隔离问题，完成了基于 Python Connect 脚本与 SSH 安全协议的稳定接入。通过针对容器化挂载引发的 `no space left on device` 问题实施精密的存储系统切片（将 Docker 与 Containerd 的 `data-root` 重定向至充沛的 `/srv` 挂载点），成功从零冷启动了计算节点。
- **后端状态监控逻辑的精准化 (Precision Monitoring Refinement)** : 在开发初期，我们捕获了一个致命的“伪全绿”故障（False Positive），即节点在网络断开（HAProxy 报 DOWN）时仍被错误判定在线。我们对 `graypulse_daemon.py` 中的解析逻辑进行了深层次重构，由原先松散的正则表达式/名称匹配机制，升级为严格的二元校验架构——只有同时满足 `row['svname']` 位于目标集群 `['gpu1', 'gpu2']` 内，且 `row['status']` 严格恒等于 `"UP"` 时，才被计为有效活跃节点。此修改为后续研究“灰色故障 (Gray Failure)”奠定了严谨的理论判定基础。
- **系统级服务的去脆弱化 (Daemonization and Audit Trail)** : 成功实现了基础设施监控进程的守护程序化。结束了前台易受中断（Ctrl+C）影响的脆弱状态，确立了稳定收集各项基线数据的测试骨架。

## 4. 下一阶段实验规划 (Next Phase: Experimental Roadmap)

根据目前健壮的“三层基座”，接下来的研究将向着模拟与治愈大规模灰色故障进入深水区。规划分为以下三个核心步骤：

### 步骤 1：故障注入 (Fault Injection via Traffic Control)

基于现有的可用空载框架，部署真实的深度学习分析模型至 Triton 仓库。利用 Linux 内核级交通控制模块 (`tc netem`) 对特定后端节点（如 `gpu1`）定向注入不规律的网络延迟波动、特定频率的静默丢包。这旨在复现真实分布式环境下的极具欺骗性的灰色故障地带。

### 步骤 2：防御策略压测与矩阵对比 (Policy Matrix Benchmarking)

在制造灰色故障后，进行系统级压力测试以捕获长尾延迟数据。将启动三个维度的横向对照实验：

1. 基线静态阈值策略 (Static Thresholding / Timeout)。
2. 经典的对冲请求策略 (Hedging Requests)。
3. 即我们提出（即将实现）的预测性灰度侦测与淘汰算法 (GrayPulse Mitigation Algorithm)。

### 步骤 3：数据汇聚与分析论证 (Data Visualization & Midterm Synthesizing)

提取压测产生的 P95/P99 延迟分位图以及系统的总吞吐量（Throughput）下潜曲线。通过深度对比三种策略，可视化其核心指标差距，以数据实证支撑我们在分布式故障治愈层面的创新性，将其汇编形成逻辑严密的中期汇报终稿。
